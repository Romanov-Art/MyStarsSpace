## Технологии: как обычно устроен Star Map (и что используют конкуренты)

### Общая архитектура, которую видно по рынку

1. **Веб‑конфигуратор** (SPA) показывает живое превью.
2. Превью делается быстро (часто в 72 dpi) для плавности интерфейса.
3. Финальный результат (300 dpi для печати) генерируется **на сервере** или в бэкенд‑пайплайне (для предсказуемого качества печати)【23cc6172-6535-4441-b540-861c88eabdee-1】【f452303e-404a-4187-97c2-18a1b09faf05-0】.

### Что касается фреймворков

- У **Under Lucky Stars** прямо подтверждён **React.js** и интеграция с **Shopify**, а также Cloudflare и Stripe【f452303e-404a-4187-97c2-18a1b09faf05-0】.
- У **Positive Prints** подтверждён интерактивный SPA‑дизайнер, но конкретный фронтенд‑фреймворк по источникам **не назван**【23cc6172-6535-4441-b540-861c88eabdee-2】.

### Какой стек именно для генерации «карты звёзд»

По доступным данным конкуренты обычно:

- используют **собственный астрономический алгоритм** (особенно там, где делается упор на «точность»)【f452303e-404a-4187-97c2-18a1b09faf05-0】;
- разводят **превью** и **финальный рендер** (72 dpi vs 300 dpi)【23cc6172-6535-4441-b540-861c88eabdee-1】;
- могут смешивать астрослой (звёзды/созвездия/планеты) и «картографический» слой (например, для love map) — у Positive Prints заметна атрибуция OSM в контексте карт【23cc6172-6535-4441-b540-861c88eabdee-2】.

## Рекомендованный стек для вашего генератора Star Map

Ниже — практичные варианты архитектуры, отражающие рыночные паттерны.

### Вариант A — «клиент‑тяжёлый» (быстрое UX + headless commerce)

- **Front‑end:** React + Next.js (SEO + SSR/SSG, удобные конфигураторы)
- **Рендер превью:** Canvas/WebGL (например, Three.js) для плавных анимаций и интерактивности
- **Финальный рендер для печати:** serverless‑функция (Node.js) → генерация **PDF/PNG 300 dpi** (например, через headless Chromium/Puppeteer)
- **Коммерция:** Shopify (Storefront API) или headless‑магазин
- **Платежи:** Stripe
- **CDN/хранение:** Cloudflare (CDN + объектное хранилище)

Плюсы:
- быстрый Time‑to‑Market за счёт экосистемы React и готовой коммерции;
- масштабируемость через serverless;
- отличный UX дизайнеров.

Минусы:
- WebGL/Canvas на мобильных потребует оптимизаций;
- генерация тяжёлых PDF может быть ресурсоёмкой.

### Вариант B — «сервер‑центричный» (максимально стабильное качество печати)

- **Front‑end:** Vue 3 + Vite (или React — по команде)
- **Рендер:** серверный SVG → PDF (контроль качества) или серверный raster (PNG) с 300 dpi
- **Back‑end:** Node.js/Express или Python (FastAPI)
- **БД:** PostgreSQL (при необходимости геоданных — PostGIS)
- **Коммерция:** Stripe Checkout / свой заказ‑пайплайн

Плюсы:
- предсказуемое качество выходных файлов;
- меньше зависимости от производительности устройства клиента.

Минусы:
- дороже по compute на каждый заказ;
- живое превью может потребовать отдельного лёгкого рендер‑слоя на клиенте.
